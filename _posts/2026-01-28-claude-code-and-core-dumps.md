While the most common use case I have for using coding agents is to...generate code...I've also found them quite useful for operational and incident response work. I experienced an interesting use case while on call over the holidays that I figured I'd document in case anyone ends up in a similar spot. In short: a rare and nearly decade-old bug in an infrequently used endpoint was filling up our disks with [core dump](https://en.wikipedia.org/wiki/Core_dump) files. Claude Code helped me track down the root cause and iterated on a fix with me. I estimate it saved me hours or a small number of days of work given my unfamiliarity with core dumps.

First I'll share the story of the incident and how Claude and I got to the bottom of it, and then I'll share some higher-level takeaways and initial impressions.
### The story
For a few days leading into the incident, we'd gotten production alerts that our web servers' disks were filling up, which is a pretty exceptional event. Not being sure if this was a one-off, we'd terminate the servers and spin up new ones (they are stateless/ephemeral, so why not?), but with time saw that this was a recurring issue. Once we realized this was not a passing issue, I explored one of the servers, and it turned out that our Docker containers were slowly accumulating gigabyte-sized core dump files. I know core dumps are created when an application aborts, and have previously used `strings` to look for hints in them that might guide me toward a potential root cause, but didn't know much about the files beyond that.

I told Claude about my first problem: I needed to get the core dumps out of the Docker container on the remote server and onto a development machine with the utilities to further investigate it. I couldn't copy the files around within the server since its disk was near full, but Claude had me covered: it remotely ran `docker cp` to copy the file out of the container, then piped that to `gzip` and send the bits over `ssh` before unzipping them on the development machine. By piping between docker, the host machine, and ssh, the files never had to be moved around on the machine itself:
```bash
for pid in <LIST OF IDs>; do
  echo "Copying core.$pid..."
  ssh user@machine "docker cp <CONTAINER ID>:/website/core.$pid - | gzip" | gunzip > core.$pid
done
```

Knowing little about core dump files, I asked Claude Code to try to figure out what the root cause was. With no other details (the core dumps were in a separate directory, so the agent initially didn't even have access to our codebase), it ran `file` on the dumps to figure out what type of file it was (a tarball/archive of other files, apparently), untarred it, and started to explore by running commands like `gdb` and `readelf`. Through `readelf`, it was able to determine that some signal had fired, but needed to dig more to determine which signal was fired.

To find the signal, Claude Code asked for permission to run a script to parse the core dump further:

Not knowing the specifics of core dump file format, my approval/code review process was to read the code enough to know it was all read-only operations that weren't accessing any external services to prevent [exfiltration](https://simonwillison.net/2025/Jun/16/the-lethal-trifecta/). Basically, I was reviewing code for "is it only going to operate on the files locally and is it not going to modify those files?" After I approved a few of these scripts, Claude concluded that all 8 core dumps arose from a SIGABRT. It then spent ~4 minutes searching for various strings across the dumps, eventually striking gold in finding Sentry profiling data from around the time of the crash. It listed a few views related to this profiling data, and suggested that one of these views might be timing out: 1) An endpoint we've had since year one of B12 to parse RSS feeds, or 2) Some not-very-clear image processing code.

I asked Claude what evidence it had of either one, and it spent quite a few minutes asking me to approve a few more scripts that I reviewed mostly for side effects. Ultimately, in a long series of `grep`s, the agent eventually searched for the URL path of the RSS feed view, and came back with evidence that 6 of the 8 core dumps had a very specific URL that was causing trouble for the RSS feed parser. Basically: **a user had set a radio stream URL as their RSS feed, and the endpoint didn't have timeouts set properly so was just streaming the endless sounds on the radio waiting for the "RSS feed" to end so it could try to parse it**. Eventually this timed out, and gunicorn/uvicorn were sending a SIGABRT to the process before killing it, resulting in the core dumps.

This type of bug is particularly hard to identify through our normal monitoring tools like Sentry or canonical logs. The request runs for a long time without error, so there's no report of a stack trace or other issue to give you a hint. And when the request finally does get killed, it comes by way of the web server/proxy timing out and killing the process, so there's no opportunity for a log line or exception to be written to disk or sent across the network, as the request is no longer running. Any sort of slow request reporting we have relies on the request coming to a close in order to determine its timing, so we simply don't have a record of that slow request's timing. In hindsight, we're missing some recording/observability of the timeout itself, which we now know and can repair.

It's less surprising that Claude Code could implement a fix to this issue, since writing code is its bread and butter. When I prompted it for a solution, the initial fix it offered up was both overly complicated (too many try/excepts and branches for somewhat nonsensical conditions) and wouldn't work in our particular codebase (relied on `httpx` whereas we heavily rely on `requests` and only recently introduced `async`/`await`). After manually testing a `requests`-based solution, I realieed it wasn't as simple as adding a `timeout` to the request, since the problem wasn't that the connection was timing out, just that it streamed forever. I was able to iterate with Claude to come to a solution that would read small chunks of data from the URL a few kilobytes at a time, and check wallclock time in between those requests. This involved elements of the `requests` API that I hadn't previously encountered. After two rounds of feedback, I had a version I was happy with and a fix on production:

```python
def _fetch_feed(feed, timeout):
    """Fetch feed with a hard total timeout."""
    start = time.monotonic()
    response = requests.get(
        feed,
        stream=True,
        timeout=timeout,
        headers={'User-Agent': 'B12 RSS Reader'},
    )
    response.raise_for_status()

    chunks = []
    for chunk in response.iter_content(chunk_size=8192):
        if time.monotonic() - start > timeout:
            response.close()
            raise requests.exceptions.Timeout('Total timeout exceeded')
        chunks.append(chunk)

    return b''.join(chunks)
```

Of course the fix didn't address the underlying misuse of our tools, so I wanted to figure out which customer this was affecting. I asked Claude Code to help me find the customer (in vague terms), and it came back with SQL to query for the RSS feed integration that the customer set up on their website. I had to modify the SQL to have it run on our data warehouse (Claude couldn't have known we weren't running the query in production). The query Claude provided was complicated, so I removed a bunch of joins and ran it and...it found the customer! This was still a time-saving query for me: unlike data sets like our funnel, I haven't written queries to extract data about integrations in a long while (ever?), so having a starting point was a great help.

With the fix live and our team contacting the customer, I headed to bed earlier than I would have if I had to investigate the issue solo. I learned a bit about core dumps and have a newfound curiosity to learn more!
### Some takeaways
Since using Claude Code in this incident, we've used it on an additional one to pretty good effect around an esoteric one-off in how our hosting stack handles wildcard TLS certificates. The goal of this blog post wasn't to tell you all about core dumps and certificates, though. I'm sharing these stories because they expanded my understanding of where we can utilize agents in our engineering processes. Here are a few broader thoughts to that end:

* This expansion from "assists with bugs and features" to "assists in incident investigations" makes me feel like "coding agent" is not a descriptive enough term for the types of use cases you can bring one of these agents onto. I'd hesitate to start giving a single tool like Claude Code multiple categories (e.g., "It's a coding agent AND a site reliability engineering agent!"). Perhaps something that more broadly encompasses the role of the person calling out to the agent, like "software engineering agent" would help prime people in that role to bring it in to help in more aspects of their day-to-day?
* For all of the heavy lifting Claude Code did, I still felt like an active participant in the process. Claude Code doesn't run on our production servers, so I was the one who explored directories until I found out what was taking up disk space (docker volumes) and then went into the container to find the large files, and then spun up Claude Code to ask how I might stream the core dumps through ssh and through docker. When given a SQL query as a starting point, I simplified it and translated it into something that would work on our data warehouse. When shown an initial fix to the code, I iterated twice through a change of library and a change of approach to implementing the timeout. And finally, I communicated with our customer success team to reach out to the customer and packaged up a PR to have a fellow engineer review. So while it saved me time, I still felt very engaged in the problem.
* For all the anthropomorphizing you could do about Claude gaining an understanding of the problem at hand, my read of the transcript of individual tool calls afterward suggests that it found bits of evidence (a signal, a signal type, a profiling trace, an endpoint, a query parameter) and `grep`ed its way between these bits of evidence to surface and then filter through new bits of evidence. There are several steps along this chain that I didn't have the knowledge to do (identify a signal was fired, parse the file to identify which signal), and by virtue of Claude "knowing" how to parse through these blobs, it was able to jump from one string search to another to uncover its next thing to parse.
* It's hard to estimate how long this would have taken without the agent's help, but I'm sure it would be longer. I could see futzing with the full disk with files inside the docker container either taking 1+ hours to connect all the pieces to get the core dumps onto a dev machine, or me making a tradeoff to delete some core dump files so I could `docker cp` others onto the host machine and then `scp`, whereas the agent proposed and I executed a solution in minutes. My one trick at exploring the core dumps (call `strings`) didn't pan out, so I could see it being hours for me to read about the file format, learn about command-line utilities, and explore the files, and I'm not convinced I would have found the right portion of memory or been able to repeatedly look across all of the core dumps to spot the pattern of the repeating URL. Even if I saw the URL, it wasn't so obviously a "this is a stream" URL, but somehow the agent identified the URL as having streaming content. We'll never know, but my guess is after a few hours, I wouldn't have connected the dots between the multiple signals surrounded by noise. And finally, knowing the issue, writing the working solution (~20 lines of code) would have been easy to write up, but maybe an hour or two to understand all of the interactions between `request`'s `timeout` (which I'm familiar with) and the `stream` kwarg and `iter_content` method (which I'm not familiar with). In contrast, I went from problem to shipping-to-production (including running tests and a deploy) solution in about 2.5 hours.
* I don't know how things would have turned out if I was left to figure this out without a coding agent, but my strong suspicion is that it would have ended up different. It was late at night during the holidays, with a clue I didn't have as much experience how to handle, and taking too long to figure out a root cause would have required manual babysitting of servers or I might have addressed the issue differently, but with more ducktape and less lasting solution or clarity on the root cause. Forcing myself to think through the evidence in front of me and lack of clarity on how to transport and meaningfully process the core dumps, I think my "solve the incident now" instinct would be to write a script that runs periodically, checks for core dumps, copies them somewhere like S3 and alerts us, and removes them from the container. This would have been workable in the sense that the servers would have no more disk space issues, but it would have kicked the can down the road without any real understanding of the root cause.
* While I now have core dumps on my "curious things to learn about" list, the incident was the wrong time to get curious about them. Prior to this incident, I was a one-trick pony when it came to core dumps: run `strings` on them and see if you can find any hints. In this case, there are MANY strings in a gigabyte-sized dump of memory, so my trick didn't quite help. It's not that I haven't used a web search to learn about a file format or better understand how to debug something during an incident, but the ability to say "hey Claude, can you get anything meaningful from these files while I collect other logs/metrics/clues" allowed me to shift my curiosity from a stressful time-sensitive incident to a later time when I can properly read up on and more deeply understand the concept.
* While it was cool that the coding agent wrote Python programs to read program registers, investigate bytecode, and dig into what was in memory with a makeshift hex editor, I'm not actually sure those things gave it the clues it needed to identify the root cause. Ultimately the presence of some stack traces that it was attempting to send to Sentry did the trick. This is not unlike what investigation process I go through, though: when you start looking for clues, you often sift through a bunch of noise before getting to some signal. In a coding agent's case, the noise can come in the form of programs and program output it writes in seconds. Maybe one leg up we have as humans is it's more expensive for us to emit code, so we might generate less noise in the process, but I bet that means we turn over less rocks as we investigate as a result.
